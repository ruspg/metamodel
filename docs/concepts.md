# Ключевые концепции

> Справочник ключевых концепций, лежащих в основе метамодели. Каждая концепция — это архитектурное решение, определяющее как мы думаем о банке.

---

## Composable Enterprise

Банк как набор переиспользуемых **business capabilities**, из которых собираются **value streams** для запуска новых продуктов.

Вместо того чтобы строить каждый продукт с нуля, команды проверяют: какие способности уже описаны, какие системы их реализуют, какие данные доступны — и собирают новый value stream из проверенных компонентов.

Это модель, которую Gartner называет *Composable Enterprise*, а ведущие банки уже реализуют: capabilities как управляемые, повторно используемые единицы, которые можно рекомбинировать для скорости.

**Связанные сущности:** `value_stream`, `value_stream_stage`, `business_capability`, `bank_product`

---

## Data Contracts

Формализованное соглашение между производителем и потребителем данных. Самостоятельная сущность метамодели (не свойство дата-продукта и не техническая спецификация поставки).

Структура заимствована из практик надёжности сервисов:

| Аналогия | В контексте данных | Что описывает |
|----------|--------------------|---------------|
| **SLI** (Service Level Indicator) | Метрика | Измеримый факт: время готовности, полнота, количество ошибок |
| **SLO** (Service Level Objective) | Требование | Целевое значение метрики: «данные готовы к 9:00», «полнота ≥ 99%» |
| **SLA** (Service Level Agreement) | Контракт | Обязательства + правила реагирования при нарушении |

**Один дата-продукт = один дата-контракт.** Одна и та же таблица в разных системах — это разные дата-продукты с разными контрактами.

**Связанные сущности:** `data_contract`, `data_product`, `data_object`, `requirement`, `sla`, `metric`

## Сертификация

Система уровней зрелости дата-продуктов, привязанная к типу потребления. Альтернатива жёстким обязательным полям: данные без полного стандарта лучше, чем отсутствие данных — но потребитель должен знать уровень.

Три категории потребления определяют требования к сертификации:

| Категория | Пример | Требования |
|-----------|--------|------------|
| **Регуляторная отчётность** | Отчёты в ЦБ, ФНС | Высший уровень — полное соответствие регуляторным стандартам |
| **Аналитика** | Дашборды, BI, ML-модели | Средний уровень — документированное качество и линейдж |
| **Операционка** | Real-time процессы, OLTP | Базовый уровень — гарантии доставки и актуальности |

Сертификация отличается от технического уровня зрелости контракта (L1/L2/L3): уровень зрелости — это техническая полнота, сертификация — бизнес-соответствие и методологическая валидация.

Сертификат может **автоматически отзываться** при нарушении SLA.

**Связанные сущности:** `data_product`, `data_contract`, `requirement`, `sla`, `metric`

## Модель владения

Владелец в метамодели — это **организационная единица**, а не физическое лицо. При смене сотрудника владение остаётся за командой.

Ключевые принципы:

- Owner дата-продукта **приоритетнее** owner'а отдельного дата-объекта
- Объект данных без продукта **сам является продуктом** (схлопывание)
- Владелец наследуется сверху вниз (product → object), если явно не переопределён
- **Открытый контракт** (без конкретного потребителя) — поставщик публично декларирует условия

**Связанные сущности:** `data_product`, `data_contract`, `organizational_unit`, `business_role`

## Граф знаний

Метамодель — это **схема** (типы и правила). Граф знаний — это **наполненный данными экземпляр**, построенный из существующих систем банка.

```
Метамодель (схема)        →  определяет типы и правила
    ↓
Онтология (словарь)       →  именует конкретные концепции (Клиент, Счёт, Кредит)
    ↓
Граф знаний (данные)      →  реальные экземпляры, связи, атрибуты из систем
```

Граф знаний строится не с нуля, а путём **соединения того, что уже есть**: каталог компонентов знает о системах, процессный инструмент — о workflow, дата-каталог — о датасетах. Метамодель даёт схему, которая делает эти связи осмысленными и запрашиваемыми.

Целевое хранилище эволюционирует: реляционная БД для первых сотен объектов → графовая БД (Neo4j) при масштабировании до десятков тысяч.

Доступ программный: MCP-интерфейсы позволяют AI-агентам обходить граф и использовать метамодель как структурированный контекст.

**Связанные сущности:** все уровни метамодели

## Кросс-доменные сущности

Три сущности метамодели применимы к **любому** объекту, не только к данным:

| Сущность | Применимость | Пример |
|----------|-------------|--------|
| **Metric** | Любой объект метамодели | Время обработки процесса, доступность системы, полнота данных |
| **Incident** | Любой объект метамодели | Нарушение SLA процесса, отказ системы, дефект данных |
| **SLA** | Любой объект метамодели | Время готовности данных, RTO системы, SLA процесса |

Это позволяет строить единую систему мониторинга и управления инцидентами поверх всех слоёв, а не создавать изолированные системы для каждого домена.

**Связанные сущности:** `metric`, `incident`, `sla` — полиморфно связаны с `business_process`, `it_system`, `data_product`, `component` и др.

## Lineage Enforcement

Принцип: требования к данным на каждом уровне цепочки должны быть **не мягче**, чем требования конечного потребителя.

Если регуляторный отчёт требует определённого уровня качества, каждый источник данных вверх по цепочке линейджа должен соответствовать этому уровню — автоматически.

Это превращает lineage из пассивной «карты потоков» в **активный инструмент управления качеством**: нарушение на любом уровне порождает событие и эскалацию.

**Связанные сущности:** `data_product`, `data_contract`, `information_flow`, `requirement`

## Тирование: критичность процесса → критичность данных

Тирность (уровень критичности) дата-продукта **не определяется самим дата-продуктом** — она выводится из критичности бизнес-процессов, которые его потребляют.

```
business_process.criticality = "Critical"
        ↓ (потребляет данные из)
data_product.tier = max(criticality всех потребляющих процессов) = Tier 1
```

Четыре уровня критичности процессов (из практики Business Impact Analysis):
- **Critical** — остановка критически влияет на банк
- **Important** — значительное влияние
- **Operational** — умеренное влияние
- **Supporting** — минимальное влияние

Это создаёт **сквозную цепочку атрибутов через слои метамодели**: стратегические цели → процессы → данные → системы — и каждый слой наследует контекст критичности.

**Связанные сущности:** `business_process`, `data_product`, `data_contract`, `sla`, `it_system`
