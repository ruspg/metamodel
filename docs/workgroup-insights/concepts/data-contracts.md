# Data Contracts (Дата-контракты)

## Определение (по состоянию на 2025-12-10)

Самостоятельная сущность метамодели, представляющая перечень обязательств владельца данных перед потребителями. Строится «над данными» и материализует их как дата-продукт.

**Не является:**
- Свойством дата-продукта (хотя изначально так рассматривалось, 2025-12-03)
- Схемой поставки (Kafka Schema Registry, API-портал = «схемы поставки», отдельный класс)
- Движком проверки качества (DQS/Kapha/DTP — внешние системы)

## Связанные entity_kinds
- `data_contract` — сам контракт
- `data_product` — контракт материализует набор данных как продукт
- `data_object` — таблицы/топики/файлы, на которые распространяется контракт
- `data_attribute` — атрибуты с требованиями к типу, обязательности, маске, справочнику
- `requirement` — абстрактное требование (текстовое описание нормы)
- `sla` — соглашение (числовая формализация требования)
- `metric` / `incident` — индикаторы и инциденты (внешние системы, за границей контракта)

## Состав контракта (из обсуждений)
1. **Схема данных** — целевая структура (не фактическая из каталога)
2. **Требования к качеству** — полнота, достоверность, своевременность, точность
3. **SLA** — время поставки, задержка стриминга, глубина хранения, расписание снэпшотов
4. **Гарантии доставки** — at-least-once / exactly-once
5. **Документация** — описание логики формирования (ссылка на бизнес-требования)
6. **Фильтрация/детализация** — какие данные включены (стрим vs снэпшот)
7. **Условные проверки** — if-then правила, маски форматов, ссылки на справочники
8. **Метаданные потребителей** — кто использует, уровни оповещения

## Статусная модель
`draft` → `active` → `certified`
- **draft** — без жёстких требований, можно создавать без полного набора
- **active** — базовый набор обязательных требований
- **certified** — подтверждение высокого качества, «методологический лайк» (новый уровень, 2025-12-03)

## Иерархия требований (2025-12-10)
```
Требование (текст, без чисел)
  └→ Соглашение (числовая формализация над индикатором)
       └→ Индикатор (факт: метрика или результат проверки)
```
Ранее три термина (требование, правило, соглашение) смешивались.

## Классификация требований
- **Жёсткие** — невыполнение = инцидент и ремонт
- **Мягкие** — невыполнение = уведомление стейкхолдеров

## Примеры из обсуждений
- Проводки ГЛ: одинаковая структура, разная логика формирования
- Операции по клиентам брокера: фильтрация (те же колонки, не все данные)
- Потоковые данные: SLA «не реже раза в час»
- Пустой ИНН у клиента, незаполненный договор — примеры дефектов

## Эволюция
| Дата | Изменение |
|------|-----------|
| 2025-12-03 | Контракт = самостоятельная сущность (не свойство дата-продукта) |
| 2025-12-03 | Контракт ≠ схема поставки |
| 2025-12-03 | Добавлен статус «certified» |
| 2025-12-03 | Гарантии доставки — часть контракта |
| 2025-12-10 | Трёхуровневая модель: требование → соглашение → индикатор |
| 2025-12-10 | Система инцидентов вынесена за границу системы контрактов |
| 2025-12-10 | Event-driven уведомления об изменениях схем |
