# Data Catalog (Дата-каталог)

## Определение (по состоянию на 2025-12-08)

Система, являющаяся **golden record** (system of record) для двух классов объектов:
1. **Объект данных** (таблица, модель, файл, Kafka-топик, ClickHouse-таблица)
2. **Процесс поставки** (пайплайн, ETL, DAG)

Остальная информация (владельцы, SLA, качество) — агрегируется из мастер-систем.

## Связанные entity_kinds
- `data_product` — центральная сущность каталога (не таблица!)
- `data_object` — физические объекты данных
- `data_attribute` — атрибуты с метаданными для людей и AI
- `data_process` — процессы поставки данных
- `data_contract` — опциональная связка (потребители первичны)
- `metric` — метрики observability
- `source` — атрибут разметки, не self-standing объект

## Ключевые решения
- **Дата-продукт, а не таблица** — центральная сущность (сдвиг фокуса, 2025-12-08)
- **Потребители первичны** — статистика использования в каталоге независимо от контракта
- **Система-источник = атрибут**, не отдельный объект каталога
- **Единое окно** — пользователь не должен открывать DataLake, каталог и DataQuest отдельно

## Новые типы метаданных
- **Разметка для LLM/Text2SQL:** категориальные типы, диапазоны числовых значений, примеры запросов
- **Confluence-ссылки в хинтах к колонкам** — автоматическое обогащение мета
- **Признак ручных операций** — важен для комплайнса/методологии финансов

## Карточка дата-продукта (подход ФинКон)
Описание + потребители + поставщики + контракты + линейдж + качество + мониторинги + SLA + тэги критичности + статистика использования + признак ручных операций + маппинг витрины (SQL)

## Примеры
- «Количество рублёвых платежей за квартал» — задача заняла 1.5 месяца из-за отсутствия навигации
- Витрина, собираемая одним большим SELECT — финансовый кейс
- Сверка остатков с БАРС — кастомный дашборд неприменим

## Открытые вопросы
- Архитектура интеграций между каталогом, контрактами и DQ-системами
- Observability: единый хаб метрик vs встроенные дашборды в каталог
