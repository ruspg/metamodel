# Эволюция идей метамодели

Хронологический трекер изменений в понимании концепций метамодели.

---

## 2025-12-04: Старт стрима «Бизнес-архитектура» (БА)

**Первая встреча стрима.** Лид — бизнес-архитектор стрима. Продукт: «Единая метамодель бизнес-архитектуры». Еженедельные встречи (чт, 17:00). Бизнес-процесс выбран первой сущностью для детальной проработки. Метрики: AI coverage бизнес-процессов, pain index, скорость process flow.

---

## 2025-12-01: Запуск — понятийный аппарат и стримы

**Отправная точка.** Руководитель инициативы представил концепт метамодели банка. Ключевые терминологические решения:
- Разграничение **метамодель** (мета-сущности: типы и связи) vs **онтология** (корпоративная бизнес-модель данных: классы и их связи) vs **данные** (экземпляры)
- Термин «модульный банк» → **composable enterprise** (бизнес-архитектор)
- Метамодель позиционируется как активность **Enterprise Architecture**, не Data

**Стратегические use-cases:** Business Activity Monitoring (маппинг компонентов на процессы), composable enterprise (переиспользование capabilities), process mining.

**Формат работы:** 4+ стрима, core-синхронизация раз в 2-4 недели, совместные интервью со стейкхолдерами.

---

## 2025-12-03: Стрим Data Contracts — определение сущности

**Поворот:** Дата-контракт признан **самостоятельной сущностью**, а не свойством дата-продукта. Ключевые уточнения:
- Контракт ≠ схема поставки (Kafka Schema Registry / API-портал = «схемы поставки»)
- Контракт существует вне дата-платформы (Kafka-топики, API, МЧС-схемы)
- Статусная модель: **draft → active → certified** (новый уровень certified)
- Гарантии доставки (at-least-once / exactly-once) — атрибут контракта
- Связь дата-продуктов с бизнес-процессами вынесена в отдельный стрим

---

## 2025-12-08: Стрим Data Catalog — golden record

**Сдвиг фокуса:** Дата-продукт признан **центральной сущностью каталога** (не физическая таблица).

Ключевые решения:
- Каталог = **golden record** для двух классов: (1) объект данных, (2) процесс поставки
- Система-источник — **атрибут** разметки, не self-standing объект
- Потребители первичны, контракт **опционален** (расширение от предыдущего понимания)
- Новое требование: **разметка для LLM/Text2SQL агентов** (категориальные типы, диапазоны значений)

---

## 2025-12-09: Стрим «Системы и компоненты» — ИТ-слой

**Уточнение границ:**
- **Интеграция** = взаимодействие двух компонентов (сужение: просто подключение к Kafka — не интеграция)
- **Ресурс** = application layer (Kafka topic, S3 bucket); инфраструктура (кластер, брокер) — отдельный слой
- **AI-агенты** зафиксированы как стратегический потребитель метамодели (семантическая точность = машиночитаемость)
- Спецификация (что публикует продюсер) vs интеграция (факт взаимодействия) — разные вещи

---

## 2025-12-10: Архитектура системы Data Contracts — границы

**Уточнение модели:**
- **Требование ≠ Правило ≠ Соглашение** — трёхуровневое разделение (ранее смешивались): требование (текст) → соглашение (числовая формализация) → индикатор (факт)
- Система инцидентов — **вне** системы контрактов (ранее предполагалось иное)
- Граница системы: контракты **не являются движком проверки** — DQS/Kapha/DTP снаружи
- Потребители контракта — **отсутствующая ветка** в схеме (признано пробелом)
- Новая идея: **event-driven уведомления** об изменениях схем/логики источников
- Автоматический **парсинг патч-документов через LLM** (уже реализован руководитель инициативыым)
- Двухуровневая классификация требований: **жёсткие** (инцидент) vs **мягкие** (уведомление)

## 2025-12-11 (БА): Слоёный пирог и бизнес-сущность в скоупе

- Концепция **«слоёный пирог»**: метамодель → операционная модель → граф знаний / цифровой двойник → AI-агенты → продукты
- Финансы переведены из «периферии» в **ядро клиентов** (терминологические боли)
- **business_entity** добавлена в скоуп стрима БА
- Метамодель включается в **техническую стратегию банка** (лоббирование руководителя)

## 2025-12-15: Финализация определений дата-контекста

**Уточнения определений:**
- Дата-продукт = «один или несколько объектов данных» (убрано «набор данных»)
- Убрана фраза «обладающий определённым уровнем качества» — достаточно «измеримая ценность»
- Дата-продукт **должен** иметь бизнес-владельца и дата-контракт (сдвиг от «может» к нормативности)
- **SLI/SLO/SLA-фрейм** для данных: метрика = SLI, требование = SLO, контракт = SLA (Дылко)
- Жизненный цикл дата-объекта: **draft → active → archiving → archived → deleted** (deleted сохраняется в каталоге для линеджа)
- Метрики **могут существовать вне контракта** (профилирование, каунты)
- **Три стрима** из одного: бизнес-контекст, дата-контекст, дата-контракт

## 2025-12-18 (БА): Фреймворк мастер-систем и golden record

- Фреймворк выбора мастер-системы: **блокирующие критерии** (нельзя создавать/редактировать, нецелевая, жёсткий атрибутный состав)
- лид глоссария: **«не будет одной мастер-системы физически — будет golden record в едином каталоге»**
- Value stream → стадия (**не «действие»** — переименовано); VS клиентоориентированный, стадии не переиспользуемы
- Показан **Protégé** (OWL-редактор) как потенциальный инструмент
- 4 требования к ведению БП: реестр, связи (IT+данные), автоматизация, ответственность

## 2025-12-17: Кросс-доменные сущности и граница дата-процесс/интеграция

**Ключевые сдвиги:**
- **metric и sla → кросс-доменные**: выносятся из дата-контекста в общий контекст метамодели
- **Владелец на уровне дата-объекта не нужен** — наследуется от дата-продукта
- Владелец = **организационная единица**, не физлицо (при увольнении — остаётся за командой)
- **data_process как сущность первого класса**: обобщение ETL, DQ-чека, стриминговой джобы (не подтип интеграции)
- Открытый вопрос: data_process = дочерняя сущность integration? Или самостоятельная?
- Идея: **дата-продукт как «микросервис данных»** — переиспользование практик продуктовой разработки

## 2025-12-22: Общие сущности и паттерн «открытый контракт»

**Метамодельные сдвиги:**
- **metric** и **incident** перемещены в общие сущности (не дата-специфичные) — применимы к любому объекту
- «Проверка качества данных» **убрана** как отдельная сущность — это data_process
- **Owner дата-продукта приоритетнее** owner дата-объекта; объект без продукта = сам является продуктом
- **Открытый контракт** (без второй стороны): поставщик публично декларирует условия
- **Надкатегория «объект управления»** (бизнес-архитектор): абстракция над data_object, business_process, it_system
- **Класс/экземпляр для процессов**: шаблон процесса vs конкретный запуск — потенциально нужна сущность process_instance
- **Правила ID**: поставщик формирует уникальный ключ; глобально = system_name + object_id

## 2025-12-23: Упрощение ИТ-слоя — рекурсия и ресурсы

**Структурные изменения:**
- **functional_subsystem → упразднить**: заменить рекурсией it_system (ИС вкладывается в ИС)
- **Компонент ИС** переименован: определение расширено (не только API/UI, но и БД, очереди)
- **logical_resource** — явное введение: S3-бакет, Kafka-топик, БД-схема (то, что потребляет компонент; в CMDB нет)
- **Жизненный цикл компонента**: source (GitLab) → artifact (Artifactory) → running software (CMDB) — стадии не различались ранее
- **AI-агенты** — в скоуп 2026: учёт как component + business_role
- **Связь «зависит от» между it_system** — выявлена потребность (ранее не было)

## 2025-12-24: Детализация дата-контракта и сертификация

**Финализация блоков контракта:**
- Обязательно: бизнес-описание, схема, ожидания значений, время готовности, актуальность, глубина хранения
- Опционально: ссылка на код (GitLab), описание методологии
- **Один дата-продукт = один дата-контракт** (таблица в разных системах = разные продукты)
- **Методология привязана к продукту**, не к контракту
- **Время готовности ≠ актуальность данных** — два разных атрибута
- **Система сертификации** вместо жёстких обязательных полей: уровни зрелости (Дылко)
- **Lineage enforcement**: требования в источнике не мягче, чем у целевого потребителя
- **Delivery semantics** (at-least-once/exactly-once, наличие дублей) — свойство контракта
- **«Дырявые» периоды данных** — фиксируются в контракте как отдельный атрибут

## 2025-12-30: Бизнес-объекты — от плоского справочника к иерархии

**Серия из 3 сессий (руководитель инициативы + эксперт по бизнес-процессам):**
- Бизнес-объекты становятся **иерархическим деревом** со связями: клиент → {Saver, Premium, B2B}; сделка → договор → счёт → комиссия
- Value streams остаются «в бэкенде» метамодели, бизнес-процессы — «продаются» бизнесу
- **Семантический поиск через LLM** как функционал датахаба: «рублёвый платёж» → объект → таблицы → системы
- Разметка данных (целевые/нецелевые источники) ставится в **цели командам стюардов на 2026**
- Внешний драйвер: **аудит банка 2026** → описание бизнес-процессов = стратегический приоритет
- Глоссарии рисков → в датахаб; регуляторные требования ЕЦБ на глоссарии

## 2026-01-12 — 01-14: Детализация атрибутов и Entity Store

**Атрибуты дата-объекта:**
- Чётко разнесены по уровням: **объект / продукт / контракт**
- **Историчность разделена**: обязательство хранения (контракт) vs фактическая глубина данных (объект)
- Статус дата-объекта: изменение инициирует owner; `TO_ARCHIVE` → автоматическое уведомление потребителей

**Entity Store:**
- Конкретизирован как **платформенный сервис** для меты бизнес-сущностей (клиент, карта, договор)
- Только мета, не данные; ролевая модель через AD-группы

**Межсистемный линейдж:**
- Единой мастер-системы для ID не будет → **федерация** между 3-5 каталогами
- Реализации: YAML для ВДВХ, кастомная логика для PowerBI, OpenLineage

**CAP vs метамодель:**
- Контракты в CAP ≠ контракты метамодели (CAP = технические схемы поставок)
- Обнаружен gap: CAP неправильно маппит «клиентский джоб» на «бизнес-операцию»

## 2026-01-20 — 01-21: AI-агенты и объединение стримов

**AI-агенты в метамодели:**
- AI-агент = **label на component** (type=agent), не отдельная сущность
- MCP-агент аналогично (type=mcp)
- Каждый агент должен иметь **собственный service account**

**Объединение стримов:**
- Data Contracts + Data Catalog → **один стрим** (две встречи: теоретическая + практическая)
- Открыт **5-й стрим: Business Glossary** (лид: Ektov)

**Observability:**
- **Два пилота**: (1) события готовности данных (Airmon), (2) события качества (DQS/GE/Dragon)
- metric / check / expectation — **три уровня**: metric = атомарное число, check = проверка владельца контракта

## 2026-02-04: Целевая архитектура и ревью для CDO

- Задача: **склейка результатов стримов** в единую целевую архитектуру метаданных
- Порядок: **definition done** по каждому стриму → общая картина
- Ревью для CDO доменов запланировано; **брейкаут забронирован на март-апрель**
- Мастер-система по бизнес-процессам не определена (CAP vs MSX vs DataHub)
- **Data Steward role** актуализируется (Korotenko)
- Стандартный демо-кейс метамодели: **«Закрытие договора страхования по объекту залога»**

## 2026-02-25 — 03-04: Уровни зрелости и TCDO

**Контракты:**
- **Три уровня зрелости контракта**: L1 (только владелец), L2 (описание + продукты), L3 (SLI, полная поддержка)
- **delivery_contract** получает определение: схема + гарантия неизменности + механизм валидации
- Два типа валидации: (a) совместимость изменений схемы, (b) данные в рантайме
- Принцип **встроенной обзервабилити**: каждый процесс генерирует телеметрию (events)

**TCDO и тирность:**
- **Total Cost of Data Ownership** — впервые названо как метрика дата-продукта
- **Тирность данных** = максимальная тирность бизнес-процессов-потребителей
- Несоответствие тирности → сигнал к эскалации

**Организационно:**
- Понедельничная общая встреча **упразднена**; работа в мини-группах по пилотам
- Переход **от теории к практике** (пилоты)
- `metamodel_object` = общий тип для любой сущности метамодели

## 2026-03-25: Сертификация дата-продуктов

**Финализация модели сертификации:**
- **Три категории потребления**: Отчётность (регуляторика) → Аналитика → Операционка (real-time)
- Высший уровень сертификации = **соответствие регуляторным требованиям** (ЦБ, ФНС)
- Сертификация ≠ уровень зрелости DoA: DoA = техническое качество, сертификация = бизнес + регуляторика
- Автоматический **отзыв сертификата** при нарушении SLA
- Референс: **все крупнейшие банки с Data Mesh** делают сертификацию с регуляторикой на верхнем уровне

## 2026-03-12: OWL-формат и оценка мастер-систем (стрим БА)

**OWL как целевой формат:**
- Бизнес-архитектор сконвертировал метамодель из Confluence PDF в **OWL/Turtle (.ttl)** через LLM — все сущности стали OWL-классами, связи → object properties, атрибуты → data properties
- Файл открывается в онтологическом редакторе, можно скормить LLM как контекст
- Существуют **MCP-серверы для OWL** — агенты могут делать запросы по графу

**Оценка мастер-систем по бизнес-процессам:**
- CAP (~417 баллов) vs Ops Process Tool (~355 баллов) — формула с нормировкой по трудозатратам
- **Альтернативный виженн Рцесса**: CAP = не реестр бизнес-процессов, а **граф знаний для агентов** (связи через трейсы/спаны микросервисов)
- Решение **отложено** до синхронизации с Рцессом

## 2026-03-19: Два уровня онтологии и пилоты бизнес-глоссария (стрим БА)

**Два уровня — ключевая концептуальная фиксация:**
- **Мета-онтология**: абстрактные классы (бизнес-процесс, бизнес-сущность, продукт, Job Family)
- **Онтология бизнеса банка**: конкретные сущности (договор, клиент, счёт, комиссия, залог)
- Это **разные артефакты**, не должны смешиваться в одном инструменте

**Пилоты бизнес-глоссария:**
- Датахаб: ведение бизнес-терминов от финансов, Operations, Retail
- Антистор: реестр бизнес-сущностей с семантическими связями (в бэклоге)
- **ER-диаграммы** — обязательный deliverable для систем-продьюсеров при загрузке в Data Lake

**Метамодель в GitLab:**
- Единый YAML + Python-конвертеры в OWL/Turtle и Mermaid
- Merge requests как механизм обсуждения (вместо Confluence)
- Целевое хранилище: Postgres → **Neo4j** (при масштабировании)

**Риск:** параллельная AI-группа (Артур Драб) пишет YAML-нотации **вразрез** с метамоделью → нужна синхронизация

## 2026-03-26: Граф знаний и BPMN-генерация (стрим БА)

**Граф знаний как целевой продукт:**
- **Knowledge graph ≠ метамодель**: метамодель = схема, граф знаний = наполненный данными экземпляр
- Строится из **существующих систем** (CAP, инфра-каталог, дата-линейдж, реестры)
- Целевое: Postgres → **Neo4j** при десятках/сотнях тысяч объектов
- Агенты ходят по метамодели через **MCP поверх OWL**

**BPMN-генерация — первый практический use-case:**
- Команда Антифрод Operations вручную рисует процессы в Figma
- Предложение: **MCP метамодели + BPMN MCP** → автогенерация BPMN из транскриптов/Confluence/Excel

**business_entity — рекурсивный класс:**
- В OWL: `business_entity hasRelation business_entity` (ссылается сам на себя)
- Явно зафиксировано бизнес-архитектором

**Критический риск:**
- Домены уже создают **собственные YAML-фреймворки** не совместимые с метамоделью
- Стратегия: **адресная работа по доменам** (не широкое роадшоу) для предотвращения расхождений

## Хронология стрима «Бизнес-архитектура» (дополнение)

### 2025-12-25 (БА): Полный разбор мета-сущностей
- Детальный разбор **всех** мета-сущностей с примерами из глоссария
- **Шорт-лист мастер-систем**: VS→Ops Process Tool, Capability→EATool (безальтернативно), Function→нет системы, Process→Ops Process Tool/SimPlan/CAP, Entity→RDP
- Маппинг: **CAP «процесс» = наша «операция»**, CAP «шаг» = наше «действие»
- Атомарный уровень = гранулярность **API-метода**
- VS клиентоориентированный, стадии не переиспользуемы (Карімов)

### 2026-01-15 (БА): Маппинг с реальными системами
- Маппинг с **Ops Process Tool**: их «функция» = наша «операция», «чек-лист» = «действие»
- **МСХ** = кандидат на мастер по **связке** процесс↔компонент (не по реестру процессов)
- Пилотный партнёр: команда **Digital Finance** (Южаков)
- ИИ уже используется внутри команды (генерация примеров для презентации)

### 2026-01-22 (БА): RDP и process mining
- **RDP** утверждена как целевая мастер по бизнес-сущности (RDMS (non-target) нецелевой)
- Process mining вошёл в повестку; «100% автоматический дискавери невозможен»
- Семантический поиск по CAP (сервис Андрея Неведена) — метамодель уже материализуется
- Три выходных артефакта: **язык** (глоссарий), **архитектура** (мастер-системы), **инструкции** (шаблоны для команд)

### 2026-01-29 (БА): Граф-модель бизнес-процесса
- Концепция Морева: БП как **граф** (вершины=цели/статусы, рёбра=агенты: люди/ИИ/алгоритмы)
- Триада «объект + агент + цель» = бизнес-операция
- Начата формализация оценки мастер-систем: **15 критериев** с весами

### 2026-02-05 (БА): LLM-генерация графа знаний и SimpleOne
- Граф знаний **сгенерирован LLM** из метамодели (Mermaid-диаграммы по слоям)
- **SimpleOne** переквалифицирован: это мастер по IT-активам, не по процессам
- **Graph RAG** даёт больше контекста, чем текстовые чанки из векторной БД
- Осознание: бизнес-архитектура (процессы) отдельна от метамодели бизнес-сущностей (глоссарий)

### 2026-02-12 (БА): BIA-процедура и архитектура мастер-данных
- **4 категории критичности**: Critical, Important, Operational, Supporting (из BIA)
- **Двушкальная оценка**: текущее соответствие + готовность команды довести (формула)
- SimpleOne **исключён** из кандидатов; остаются CAP и Ops Process Tool
- Первая **концептуальная архитектура потоков мастер-данных** между системами

### 2026-02-19 (БА): Финализация методики оценки
- Шкалы **0/3/5** для каждого критерия с конкретными примерами
- Визуальное моделирование — **отдельный класс систем** (не мастер-система реестра)
- Два **воркшопа** запланированы: с CAP и Ops Process Tool (очная верификация)

### 2026-03-05 (БА): Трудозатраты, тирность и ветвления
- Колонка «готовность» → **трудозатраты (person-days)** с пониженным весом
- **Тирность данных = критичность процессов** (формальная связь через слои метамодели)
- **Бизнес-операция с ветвлениями** (variative outcomes) — структурное изменение; CAP планирует новый UI
- Метамодель → **Git** в code-based формате (OWL/Mermaid ER) для машиночитаемости и RAG
- Контекст для AI-агентов: `it_system → component → git_repo + business_operation + data_entity + api_contract`
