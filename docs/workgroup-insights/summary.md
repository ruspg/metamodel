# Метамодель банка — сводное понимание концепции

> Синтез 41 транскрипции встреч (декабрь 2025 — март 2026)

---

## 1. Что такое метамодель банка

**Метамодель** — формальная система типов сущностей и связей между ними, описывающая банк на пяти уровнях: от стратегических целей до инфраструктуры. Это **«словарь словарей»**: не конкретные процессы или системы, а правила того, *какие виды* объектов существуют и *как* они связаны.

**Онтология** (корпоративная) — конкретные экземпляры классов метамодели: конкретные домены (Retail, CIB), конкретные процессы, системы — но не единичные данные (не миллионы клиентов).

**Ключевая метафора:** Метамодель = чертёж чертежей. Онтология = конкретный чертёж. Данные = построенное здание.

## 2. Пять уровней метамодели

```
┌─────────────────────────────────────────────────────────┐
│  STRATEGIC VIEW                                          │
│  Goal → Business Domain → Product → Org Unit → Segment  │
├─────────────────────────────────────────────────────────┤
│  BUSINESS DETAILS                                        │
│  Value → Value Stream → VS Stage → Capability            │
│  Process → Operation → Action → Business Entity          │
├─────────────────────────────────────────────────────────┤
│  SOLUTION DETAILS                                        │
│  IT System → Component → Integration → Info Flow → API   │
├─────────────────────────────────────────────────────────┤
│  COMPONENT DETAILS                                       │
│  Vendor Product → Software Component → API Method        │
├─────────────────────────────────────────────────────────┤
│  INFRASTRUCTURE DETAILS                                  │
│  Logical Resource → Infra Resource → Component Instance  │
└─────────────────────────────────────────────────────────┘

+ DATA LAYER (сквозной):
  Data Product → Data Object → Data Attribute
  Data Contract → Data Process → Delivery Contract

+ CROSS-DOMAIN:
  Metric, SLA, Requirement, Incident, Event
```

## 3. Ключевые концепции и их эволюция

### 3.1 Composable Enterprise
Банк как набор переиспользуемых **business capabilities**, из которых собираются **value streams** для запуска новых продуктов. При появлении нового продукта: (1) проверяем какие capabilities есть, (2) переиспользуем, (3) достраиваем. **Use-case:** быстрый time-to-market.

### 3.2 Data Contract (Дата-контракт)
**Самостоятельная сущность** (не свойство дата-продукта). Перечень обязательств владельца данных перед потребителями.

**Состав:** бизнес-описание + схема + ожидания значений + время готовности + актуальность + глубина хранения + гарантии доставки (at-least-once/exactly-once) + «дырявые» периоды.

**Правило:** один дата-продукт = один дата-контракт. Таблица в разных системах = разные продукты.

**Контракт ≠ схема поставки** (Kafka Schema Registry = «схема поставки», delivery_contract).

**Иерархия требований:** Требование (текст) → Соглашение/SLO (числовая формализация) → Индикатор/SLI (факт/метрика). Аналогия SLI/SLO/SLA из микросервисной архитектуры.

**Три уровня зрелости:** L1 (владелец), L2 (описание + продукты), L3 (SLI, полная поддержка).

### 3.3 Data Product (Дата-продукт)
**Центральная сущность каталога** (не таблица). Автономный набор ≥1 дата-объектов с измеримой ценностью. Должен иметь бизнес-владельца (организационную единицу) и дата-контракт.

**Жизненный цикл:** draft → active → archiving → archived → deleted (deleted сохраняется для линеджа).

**Сертификация:** уровни зрелости привязаны к типу потребления:
- Отчётность/регуляторика (высший — ЦБ, ФНС)
- Аналитика
- Операционка (real-time)

**Тирность** = максимальная тирность бизнес-процессов-потребителей (не продюсера). **TCDO** (Total Cost of Data Ownership) — метрика стоимости дата-продукта.

### 3.4 Systems & Components
- **IT System** — группа компонентов; допускается **рекурсия** (ИС в ИС, заменяя functional_subsystem)
- **Component** — то, что команды разрабатывают и деплоят (микросервис, batch, БД)
- **Integration** = взаимодействие двух компонентов (нужен и продюсер, и потребитель)
- **Logical Resource** (новая сущность) = то, что потребляет компонент (Kafka topic, S3 bucket, БД-схема)
- **AI-агент** = label на component (type=agent), не отдельная entity kind

### 3.5 Metric, SLA, Incident — кросс-доменные
Применимы к **любому** объекту метамодели (не только данным): бизнес-процессу, IT-системе, компоненту. Вынесены в общий контекст.

### 3.6 Data Process
**Сущность первого класса** (заменяет «поток загрузки данных»). Обобщение ETL/ELT, DQ-чека, стриминговой джобы. Имеет входные/выходные данные, рантайм (постоянный или периодический).

### 3.7 Entity Store
Платформенный сервис для **меты бизнес-сущностей** (клиент, карта, договор). Хранит только мета (поля, типы), не данные. Ролевая модель через AD-группы.

### 3.8 Lineage Enforcement
Требования в источнике не мягче, чем у целевого потребителя — **автоматическая пропагация** по цепочке линеджа.

## 4. Организация работы

### Стримы (рабочие группы)
1. **Бизнес-архитектура** — сущности + связи бизнес-уровня
2. **Системы и компоненты** — ИТ-слой
3. **Дата-контекст + дата-контракты** — объединены в один стрим
4. **Observability** — хаб событий, телеметрия (пилоты: Airmon, DQS)
5. **Business Glossary** — бизнес-термины и маппинг на данные

### Состав рабочей группы
Кросс-функциональная команда из 15+ специалистов: бизнес-архитекторы, дата-инженеры, аналитики, разработчики. Каждый стрим имеет выделенного лида.

### Формат и артефакты
- Метамодель v1.0 → **GitLab-репозиторий** (YAML + Mermaid/OWL)
- MCP-интерфейс для программного доступа к сущностям
- Стандартный демо-кейс: «Закрытие договора страхования по объекту залога»
- Ревью для CDO доменов + общебанковский брейкаут (март-апрель 2026)

## 5. Два уровня онтологии (ключевая концептуальная фиксация, март 2026)

Зафиксировано разделение на два артефакта:
- **Мета-онтология** — абстрактные классы: бизнес-процесс, бизнес-сущность, продукт, Job Family и т.д. Хранится в Git (YAML + OWL/Turtle)
- **Онтология бизнеса банка** — конкретные сущности: договор, клиент, счёт, комиссия, залог. Экземпляры/подклассы `business_entity`

Метамодель = схема. **Граф знаний** = наполненный данными экземпляр, строящийся из существующих систем (CAP, инфра-каталог, дата-линейдж). Целевое хранилище: Postgres → Neo4j.

## 6. Открытые вопросы (на конец марта 2026)

1. **Мастер-система по бизнес-процессам** — CAP vs Ops Process Tool; запарковано до нового владельца CAPа и виженна Рцесса
2. **Граница data_process / integration** — окончательная модель связи не зафиксирована
3. **Бизнес-глоссарий** — пилоты запущены (Датахаб + Антистор), владелец не определён
4. **Межсистемный линейдж** — стандарт ID отсутствует; решение = федерация каталогов
5. **Сертификация** — драфт уровней готовится Красовой
6. **Риск параллельных онтологий** — домены создают собственные YAML-фреймворки, несовместимые с метамоделью
7. **Синхронизация с AI-группой** (Артур Драб) — работают вразрез с метамоделью

## 6. Стратегические use-cases метамодели

| Use-case | Описание | Связанные сущности |
|----------|----------|-------------------|
| **Composable Enterprise** | Быстрый запуск продуктов через переиспользование capabilities | business_capability, value_stream, bank_product |
| **Business Activity Monitoring** | Сквозной мониторинг прохождения продуктов через сервисы | component, business_process, metric |
| **Process Mining** | Замер step time, process effectiveness | business_process, business_operation, metric |
| **Цифровой двойник банка** | Data Ready стратегия, AI-агенты, «что если» аналитика | все уровни метамодели |
| **Total Cost of Data Ownership** | Стоимостная оценка дата-продуктов | data_product, business_process, metric |
| **AI-агенты** | Контекст для LLM: семантический поиск, Text2SQL | data_object, data_attribute, business_entity |
| **Сертификация данных** | Уровни соответствия регуляторным требованиям | data_product, data_contract, requirement |
| **Lineage Enforcement** | Автоматическая пропагация требований по цепочке | data_product, data_contract, information_flow |
