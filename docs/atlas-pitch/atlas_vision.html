Что такое Atlas — за 60 секунд

Не ещё одна система документирования. Живая, машиночитаемая, валидируемая карта банка.

Метамодель = строительный кодекс

Формальные правила: какие типы сущностей существуют в банке (процессы, системы, продукты, данные), какие связи между ними допустимы, какие ограничения целостности обязательны. YAML-авторинг, Git-версионирование, автовалидация.

Граф знаний = подробная карта города

Конкретные экземпляры: «Кредитный конвейер» → реализован через loan-origination-service → потребляет данные из scoring-engine → принадлежит домену Retail. Все связи — типизированные, запрашиваемые, валидируемые.

Метамодель для банка — то же, что Google Knowledge Graph для поиска: структура, которая делает всё остальное умнее. Без неё AI — это быстрый поиск по Confluence. С ней — архитектурный навигатор, который понимает контекст.

Существующие системы не заменяются — они встраиваются

Каждая система банка уже закрывает свой кусок знаний. Atlas — это семантический слой поверх, который соединяет фрагменты в единую картину. Никаких миграций.

Стратегический
Ops Process Tool (value streams) HR-система (оргструктура) нет сквозных связей с IT Atlas: goal → domain → product → org_unit
Бизнес
Ops Process Tool (бизнес-процессы) Бизнес-глоссарий процессы не связаны с системами Atlas: process → operation → it_system
Решения
SimpleOne (ИТ-системы — мастер) CAP (операции ↔ компоненты) design-time, не runtime Atlas: it_system → component → api → data_product
Компоненты
MSH (компоненты — мастер) Service Mesh нет связи с бизнес-функциями Atlas: component → capability → value_stream
Инфраструктура
Мониторинг (Grafana) Cloud Management алерт без бизнес-контекста Atlas: infra → component → process → product
Данные
DataHub (метаданные, lineage) Data Owner App (дата-продукты) lineage без семантики Atlas: data_product → sli/slo → contract → owner

Ключевой инсайт

Каждая система — источник фактов. SimpleOne знает про ИТ-системы, Ops Process Tool — про процессы, DataHub — про lineage. Но ни одна не знает, как ИТ-система связана с процессом, который создаёт дата-продукт для регуляторного отчёта. Atlas — это недостающие связи.

Техническая реализация

Коннекторы к SimpleOne, MSH, Ops Process Tool, CAP, DataHub, Data Owner App через API/CDC. Маппинг на онтологию через relation_catalog. MCP-сервер для AI-агентов. OWL/Turtle-экспорт. Всё в Git — код-ревью, версионирование, CI/CD.

AI меняет команды. Знания не должны пропасть.

Конвергенция ролей — неизбежна и полезна. Но когда 5 ролей сливаются в одну, объём неявных знаний на человека растёт кратно. Онтология — страховка от потери контекста.

Решение: граф знаний как память организации

Atlas фиксирует не код, а контекст: почему принято решение, какие процессы связаны, где зависимости, кто владелец.

AI-агент с доступом к графу: принимает требование → находит затронутые сервисы → генерирует код с учётом контрактов данных → создаёт тесты из бизнес-правил онтологии → предлагает деплоймент-план. Каждый шаг валидируется против метамодели.

🏦 Goldman Sachs — Legend/PURE

🌐 JPMorgan Chase

Внутренний knowledge graph: системы, данные, процессы, оргструктура — всё связано. Impact analysis и regulatory reporting at scale на 300,000+ сотрудников и тысячи систем. Каждый элемент архитектуры — в графе.

🇪🇺 ING Bank

Enterprise ontology как основа платформенной трансформации. Доказательство: европейский банк может принять ontology-driven architecture и получить измеримые улучшения в time-to-market и операционной прозрачности.

📊 Microsoft

Ни один из этих примеров — не стартап, экспериментирующий с графами. Это крупнейшие финансовые организации мира, вложившие сотни миллионов долларов в формализацию знаний. Вопрос не «стоит ли это делать», а «как быстро мы начнём».
Roadmap: пилот на домене Operations

Не big bang. Начинаем с одного домена, строим граф из существующих реестров, проводим серию измеримых экспериментов.

Фаза 1 · 0–6 месяцев · Пилот Operations

Разметка. Интеграция. Эксперименты.

Разметка баз знаний на онтологию Atlas, интеграция реестров в единый граф, API + MCP, серия пилотов.

  • Разметка сущностей: SimpleOne (ИТ-системы), MSH (компоненты), Ops Process Tool (процессы), CAP (связи операций ↔ компонентов), DataHub (lineage), Data Owner App (дата-продукты)
  • Интеграция в единый граф знаний
  • API-слой (REST/GraphQL) + MCP-сервер
  • EXP-1: Auto-discovery фичей из графа — обнаружение скрытых связей между сущностями
  • EXP-2: Impact analysis через граф — «что затронет изменение системы X»
  • EXP-3: A/B-тест LLM: с графом vs без (методология Sequeda)
  • EXP-4: Автогенерация ответов на регуляторные запросы
Фаза 2 · 6–12 месяцев

Три домена. AI-агенты. Регуляторика.

Масштабирование на Operations + Retail + CIB. AI-агенты в продакшене.

  • MCP-сервер для AI-агентов в продакшене
  • SHACL-валидация всех изменений графа
  • Регуляторные запросы из графа — стандартный процесс
  • Онбординг новых сотрудников: дни вместо месяцев
  • Proposal-based workflow для изменений
Фаза 3 · 12–24 месяца

Платформа. Полный граф. Цифровой двойник.

Все домены. Runtime-синхронизация. Предиктивный impact analysis.

  • Полный корпоративный граф знаний
  • Auto-discovery из Service Mesh / APM
  • Два семантических слоя: метрики + знания
  • What-if симуляции изменений
  • Self-documenting architecture
